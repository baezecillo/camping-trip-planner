### Frontend Iteration 1 (Sat Jul 18 23:05:49 EDT 2026)
```
You are implementing the frontend for a project called
"camping-trip-planner". Read docs/step1-base-system/02-design.md in this
repository first — it defines the two-screen flow, the component
breakdown, and the API contract you must call. Follow it exactly.

Scope: frontend only. Do not modify backend/ or docker-compose.yml in
this task. The backend is already implemented and exposes the API
described in 02-design.md at http://localhost:8080.

Build the frontend/ directory as a React + Vite single-page app:

1. Project setup: Vite + React (JavaScript, not TypeScript), with a
   frontend/.env.example placeholder for VITE_GOOGLE_MAPS_KEY (the real
   frontend/.env is git-ignored; do not commit real keys).

2. All API calls must use `credentials: 'include'` on fetch (or the
   equivalent on whatever HTTP client you choose) so the session cookie
   is sent — auth is session-cookie based, not token based. Assume the
   backend's CORS policy will be updated separately to allow the dev
   server's origin; you do not need to configure CORS yourself.

3. LoginPage / RegisterPage: simple forms calling POST /api/auth/login
   and POST /api/auth/register. On successful login, transition to the
   TripPlanner view.

4. TripPlanner (parent component): on mount, call GET /api/trips/current.
   - If 404, render SearchScreen.
   - Otherwise, render TripDetailScreen with the returned trip + checklist.

5. SearchScreen: From, Where, When (start/end date) inputs, and a Go
   button. On submit, POST /api/trips with those values, then transition
   to TripDetailScreen using the response.

6. TripDetailScreen, composed of:
   - WrapUpButton at the top: calls DELETE /api/trips/current, then
     transitions back to SearchScreen on success.
   - RouteMap: an <iframe> using the Google Maps Embed API in directions
     mode, e.g.
     `https://www.google.com/maps/embed/v1/directions?key=${apiKey}&origin=${origin}&destination=${destination}&mode=driving`
     reading the API key from import.meta.env.VITE_GOOGLE_MAPS_KEY, and
     origin/destination from the trip data (URL-encode both values).
   - Countdown: displays the trip's daysUntilStart value as-is (it's
     already computed server-side — do not recompute it client-side).
   - Checklist: renders items grouped by category. Each item is a
     checkbox reflecting isPacked; toggling it calls
     PATCH /api/checklist/{id} with the new value, and updates local
     state only after a successful response (don't assume success).

7. Basic component tests using Vitest + React Testing Library (Vite's
   standard testing pairing) covering at minimum:
   - SearchScreen renders all four inputs and the Go button
   - Submitting SearchScreen calls the trips POST endpoint (mock fetch)
   - TripDetailScreen renders the countdown value and all checklist items
     grouped by category
   - Clicking a checklist checkbox calls PATCH with the item's id (mock
     fetch)
   - Clicking Wrap Up calls DELETE (mock fetch)

After implementing, run `npm install`, then `npm test -- --run` and
`npm run build` yourself inside frontend/, and fix any failures before
finishing your turn.
```

### Single-shot task (Sat Jul 18 23:34:37 EDT 2026)
```
You are fixing a CORS configuration gap in the Spring Boot backend for
"camping-trip-planner" (backend/). The frontend runs on Vite's dev server
at http://localhost:5173 and calls the backend's API using
credentials: 'include' (session-cookie auth), but the backend currently
has no CORS policy, so browser requests from the frontend will fail even
though curl-based testing worked fine (curl isn't subject to CORS).

Scope: backend only.

Requirements:
1. Configure CORS to allow origin http://localhost:5173 specifically —
   do not use a wildcard ("*"), since wildcard origins are incompatible
   with allowCredentials(true) per the CORS spec and browsers will reject
   it.
2. Allow credentials (cookies) to be sent cross-origin.
3. Allow methods: GET, POST, PATCH, DELETE, OPTIONS.
4. Allow the Content-Type header at minimum.
5. Apply this policy to /api/** and make sure it's wired into the
   existing Spring Security filter chain correctly (a CorsConfigurationSource
   bean alone is not enough if Spring Security's own filter chain doesn't
   reference it — verify http.cors(...) is actually enabled in the
   SecurityFilterChain, not just a standalone WebMvcConfigurer).
6. Do not change any existing authentication/authorization logic — this
   is additive, not a security rewrite.

After making the change, run `./gradlew build test` yourself inside
backend/ and confirm it still passes (this change should not break any
existing test), then finish your turn.
```

API contract, the graceful-degradation cases, and the test plan. Follow

Implement, inside backend/:

1. A new service (e.g. WeatherService) that:
   a. Geocodes the trip's destination string via
      https://geocoding-api.open-meteo.com/v1/search?name={destination}&count=1
   b. Calls https://api.open-meteo.com/v1/forecast with the resulting
      lat/lng, daily=weather_code,temperature_2m_max,temperature_2m_min,precipitation_probability_max,
      forecast_days=16, timezone=auto
   c. Finds the entry in the response's daily.time array matching the
      trip's startDate.

2. A new endpoint: GET /api/trips/current/weather, requiring the same
   session auth as other trip endpoints (401 if not logged in, 404 if no
   active trip), returning the exact response shapes defined in
   02-design.md's API Contract section — including the unified
   {"available": false} shape for ALL THREE degradation cases (beyond
   16-day horizon, zero geocoding results, forecast API failure). None of
   these three cases should ever produce a 5xx response.

3. Both external HTTP calls must have an explicit timeout (5 seconds) —
   do not let a slow external API hang the request.

4. Write tests (mocking the external HTTP calls — do not hit the live
   Open-Meteo APIs in the test suite) covering exactly the four scenarios
   listed in 02-design.md's Test Plan section:
   - Trip within 16 days returns a real forecast
   - Trip beyond 16 days returns {"available": false}
   - Geocoding returning zero results returns {"available": false}
   - A simulated forecast-API failure returns {"available": false}, not
     a 500

After implementing, run `./gradlew build test` yourself inside backend/
and fix any compilation or test failures before finishing your turn.
